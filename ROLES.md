# ロール定義

このファイルは、指示の混線を防ぐために、プロジェクト内の役割を明確化するための定義です。

※JSTQB内での定義準拠（本プロジェクト向けに用語を整理）

## 対象ロール

1. Tester（テスト実行者）
2. TestDesigner（テスト設計者）
3. TestAutomationEngineer（テスト自動化設計者）
4. Developer（開発者）

## 1) Tester（テスト実行者）

- 目的: 定義済みの手順を正確に実行し、事実ベースで結果を記録する
- 主な作業:
  - `test/DevTest/REGRESSION-UI-CHECKLIST.md` に沿った手動確認
  - headed 実行で画面挙動を確認
  - 失敗時エビデンス（スクリーンショット/動画/trace）の保存
- 成果物:
  - 実施ログ（実施日、結果、再現手順、証跡パス）
- 判断範囲:
  - バグの疑いを報告するまで（根本原因の断定はしない）

## 2) TestDesigner（テスト設計者）

- 目的: 仕様とリスクに基づいて、何をテストすべきかを定義する
- 主な作業:
  - 観点整理（正常/異常/境界/状態遷移）
  - 手順書・期待値の作成/更新
  - ケースの優先度付け（回帰対象の選定）
- 成果物:
  - 手動テスト項目、期待値、実施順序
  - `test/DevTest/REGRESSION-UI-CHECKLIST.md` 更新案
- 判断範囲:
  - テスト観点とカバレッジの定義

## 3) TestAutomationEngineer（テスト自動化設計者）

- 目的: 手動テストを自動化可能な形に落とし込み、継続実行できる状態にする
- 主な作業:
  - Playwright/pytest の自動テスト設計
  - セレクタ戦略、待機戦略、安定化設計
  - 証跡出力方針（screenshot/video/trace）設計
- 成果物:
  - `test/DevTest/test_frontend.py` などの自動テストコード
  - `test/DevTest/README.md` の実行手順更新
- 判断範囲:
  - 自動化可否、実行モード（headed/headless）、証跡要件

## 4) Developer（開発者）

- 目的: 不具合修正と機能実装を行い、品質を担保する
- 主な作業:
  - 原因調査とコード修正
  - API/UIの整合性確保
  - 修正後の再発防止（テストケース追加）
- 成果物:
  - 実装コード、修正内容、変更ドキュメント
- 判断範囲:
  - 実装方法と技術的トレードオフ

## 受け渡しルール

- 実行者 -> 設計者:
  - 事象、再現手順、証跡を渡す
- 設計者 -> 自動化設計者:
  - 自動化対象ケースと期待値を渡す
- 自動化設計者 -> 開発者:
  - 落ちるテストと失敗条件を渡す
- 開発者 -> 実行者:
  - 修正済みブランチと再確認観点を渡す

## 運用上の原則

- 教材/レビュー用途の `DevTest` は headed 実行を基本とする
- CI/定期実行は headless を許容するが、失敗時証跡は必ず保存する
- 役割を兼務する場合でも、どのロールとして判断したかを明記する
