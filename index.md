# 実行者から設計者への育成ガイド

## 背景

暗号資産取引プラットフォーム案件という、極めて正確性とセキュリティが求められる現場でのPL業務、お疲れ様です。

「実行者は指示通りに動けるが、なぜそのテストが必要なのか（設計）がわからない」という壁は、多くのPM/PLが直面する課題ですね。特に2次請けという立場だと、工数管理も厳しく、教育コストがダイレクトに納期を圧迫するジレンマもよくわかります。

設計者に引き上げるためのトレーニングを「型」化するためのアプローチを整理しました。

## 1. なぜ「実行」から「設計」への移行が難しいのか？

実行者は「点（テストケース）」を見ていますが、設計者は「面（仕様書や業務フロー）」を見ています。この視点のギャップを埋めるのがトレーニングの目的です。

- 実行者の視点：「Aを入力してBになればOK」という手順の消化
- 設計者の視点：「もしCという例外が起きたら？」「この機能は別のD機能に影響しないか？」というリスクの網羅

## 2. 推奨されるトレーニング方法論：逆引きアプローチ

いきなり「テスト技法（境界値分析など）」の座学から入ると、現場で使いこなせず挫折します。暗号資産取引のような金融系案件なら、以下のステップが有効です。

### STEP 1：既存ケースの「意図」を言語化させる（リバース設計）

新しく設計させる前に、「今自分が実行しているテストケースが、なぜ作られたのか」を既存の仕様書と紐付けさせます。

- ワーク：過去のテスト項目を数件渡し、「この項目は仕様書のどの1行を担保するためのものか？」を書き出させる
- 効果：仕様書を「読む対象」から「設計の根拠」に変える意識付けになります

### STEP 2：テストタイプと観点の整理（マインドマップ）

「何をテストすべきか」という「テスト観点」の引き出しを増やします。

- ワーク：「ログイン機能」というお題に対し、正常系だけでなく、異常系（パスワード間違い、ロック）、境界値（文字数制限）、非機能（通信断、複数端末ログイン）などの観点を箇条書きさせる
- ツール活用：マインドマップ形式で、思考を広げる訓練が効果的です

### STEP 3：不具合表からのフィードバック

過去に出たバグ（不具合票）を教材にします。

- ワーク：「このバグは、設計時のどの考慮が漏れていたから防げなかったのか？」を議論する
- 効果：「設計漏れ＝現場の混乱」という実感を持ち、危機意識を高めます

### STEP 4：仮PJ定義書から不足要素を探す（欠陥発見演習）

わざと不完全な仕様書や要件定義書を用意し、「足りない要素」を見つけ出させる訓練です。

- ワーク：仮のプロジェクト定義書を渡し、「このまま開発を進めたらどんな問題が起きるか？」「どの観点が抜けているか？」をリストアップさせる
- 効果：設計者視点での「網羅性」と「リスク予測」のトレーニングになります
- サンプル：[example-pj-definition.md](example-pj-definition.md) を参照

主な確認ポイント：

- 入力項目のバリデーション（必須チェック、形式チェック、桁数制限など）
- エラーメッセージの表示位置・内容
- ボタンの活性/非活性制御
- 画面遷移とステータス表示の整合性
- ローディング表示のタイミング

## 3. 教材に盛り込むべき「フロントエンドUAT特有」の要素

フロントエンドのUATだからこそ、以下の観点をトレーニングに組み込むと即戦力化しやすいです。

| 分類               | トレーニングすべき観点                                                                |
| ------------------ | ------------------------------------------------------------------------------------- |
| 入力バリデーション | 必須チェック、形式チェック（アドレス、金額）、範囲チェック（最小/最大値）、小数点桁数 |
| エラー表示         | エラーメッセージの文言、表示位置（項目直下/画面上部）、複数エラー時の優先順位         |
| 状態管理           | ボタンの活性/非活性、ローディング中の操作制御、処理中の二重送信防止                   |
| 画面遷移           | 申請後の画面遷移、キャンセル時の戻り先、ブラウザバック時の挙動                        |
| 表示内容           | 手数料の表示、受取額の計算表示、履歴一覧の表示項目とソート                            |

## 4. PLとしての運用戦略：スモールステップ化

他のPLから「一気に引き上げるのは難しい」という声が出ているのは、「設計＝ゼロから100まで全部書くこと」を想定しているからです。

以下の段階を踏む提案をしてみてはいかがでしょうか。

- レベル1（追記）：既存ケースへの「追加観点」を1〜2件提案させる
- レベル2（一部設計）：機能の一部（例：バリデーションチェックのみ）を設計させる
- レベル3（ペア設計）：PLと一緒に30分だけ画面を見ながら設計し、残りを任せる

## 5. テスト設計の3段階プロセス

テスト設計は、以下の3つの段階に分けて進めると、作業が整理され、抜け漏れを防げます。
各段階で求められる思考や観点が異なるため、段階ごとに習得していくアプローチが有効です。

### 5.1 因子水準出し

仕様書から「何をテストすべきか」を洗い出す段階。

- テストすべき要素（因子）を特定する
- 各因子に対して、どんな値のパターン（水準）があるかを網羅する
- 詳細：[01_factor-level.md](01_factor-level.md)

### 5.2 パターン表埋め

因子と水準を組み合わせて、具体的なテストパターンを作る段階。

- 正常系、異常系、境界値などのパターンを表形式で整理する
- 組み合わせの網羅性を確認し、省略できる部分を判断する
- 詳細：[02_pattern-table.md](02_pattern-table.md)

### 5.3 手順期待値出し

各テストパターンを実際に実行可能な形に落とし込む段階。

- 具体的な操作手順を記述する
- 期待される結果（画面表示、エラーメッセージなど）を明記する
- 詳細：[03_procedure-expected.md](03_procedure-expected.md)

この3段階を順番に進めることで、「何を」「どう組み合わせて」「どう確認するか」が明確になります。
